Why？

为了实现cuda编程类内容的自动化，来学习一下这个脚本类的东西

vim main.c

gcc main.c -o main 这条命令负责调用编译工具链，将源文件 main.c 编译、链接为可执行文件 main

OK先编辑一个简单的Makefile程序

vim Makefile

然后在wsl中执行一下这个

因为之前已经gcc编译过了，因此显示的是make: ‘main’ is up to date.

也就是说即使重新编译、链接得到一个新的 main，它与旧的 main 也不会存在任何的不同，所以Makefile没有执行任何的步骤。

修改一下main.c的内容再进行一下make

Makefile会自动地执行 gcc main.c -o main，以得到新的可执行文件 main。从结果来看，代码中的修改确实反应到了可执行文件上。

Makefile三要素

两行语句可以分为三部分，分别是目标（target）、依赖（prerequisite）和执行语句（recipe）

上面的例子中，可执行文件 main 就是我们想要得到的目标，而 main 的生成依赖于 main.c，所以将 main.c 填写在依赖的位置。在发现目标文件不存在，或依赖的文件有所修改后，Makefile 就会执行下方的执行语句，其任务通常是生成目标文件。

我们在文件夹中添加新的源文件 bar.c，并将 main.c 重命名为 entry.c，这时再执行 make 会得到怎样的结果呢？

修改之后在终端进行执行make

由于找不到依赖文件 main.c， make 停止了执行。解决问题的方法有两种，简单粗暴的做法是：直接根据新的文件命名修改 Makefile 文件

由于主函数调用了 bar.c 中定义的函数，所以在编译时我们需要将 bar.c 一起编译、链接到可执行文件里，同时别忘了把它加进依赖中。修改好后回到终端重新执行 make

如果一直这样去做的话，后面每次加源文件我们都需要更改Makefile文件，这样我们的自动化的意义基本上属于是灭有

变量和通配符和wildcard函数

我们源文件都是以.c结尾因此我们可以用这种模式匹配所有源文件，在 Makefile 中我们可以使用 wildcard 函数（wildcard function）来达到这一目的。
